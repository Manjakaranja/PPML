# S3 Data Lake Storage

This section documents the S3 storage used in the FlyOnTime project.

S3 is used as our cloud data lake. It stores training datasets, MLflow artifacts, inference inputs and outputs, logs, presentation material, and demo assets.

AWS S3 console:  
https://eu-north-1.console.aws.amazon.com/s3/buckets/

---

## Buckets overview

### `ppml-mlflow`

This bucket is used as the **MLflow artifact store**.

It stores the artifacts generated during model training, including:

- trained model files;
- MLflow run artifacts;
- preprocessing artifacts when logged;
- model registry artifacts used later by FastAPI for inference.

This bucket is mainly used by the MLflow tracking server.

---

### `ppml2026`

This bucket is the main FlyOnTime project data lake.

It contains datasets, app assets, raw inference outputs, processed inference files, presentations, and demo recordings.

#### Structure

```text
ppml2026/
├── datasets/
├── image/
├── presentations/
├── raw/
├── processed/
└── screen_records/
```

---

## `datasets/`

This folder contains the datasets used for:

- exploratory data analysis;
- feature engineering;
- model training;
- inference schema reference.

Examples:

```text
datasets/
├── SignofFlightsDataset_20260416_233018_CLEAN.csv
└── df_train_final.parquet
```

The file `SignofFlightsDataset_20260416_233018_CLEAN.csv` is used as the main training source for the XGBoost classifier.

After feature engineering, the processed training dataframe is saved locally as:

```text
data/df_train_final.parquet
```

This file is then used by the XGBoost regressor training script and can also be stored in S3 to show the public version of the feature-engineered dataset.

## `raw/`

The `raw/` folder stores raw outputs generated by the on-demand API pipeline.

It corresponds to the **Bronze layer** of the data lake.

Each user request is stored in a dedicated folder to support traceability, debugging, and multi-user compatibility.

### Structure

```text
raw/
└── YYYY-MM-DD/
    └── request_id/
        ├── SignoffFlightsDataset_Single_<request_id>.parquet
        ├── API_Single_ERR.log
        └── flight_request_status.json
```

Example:

```text
raw/
└── 2026-04-22/
    └── requete_AF1234_20260422_170113/
        ├── SignoffFlightsDataset_Single_requete_AF1234_20260422_170113.parquet
        ├── API_Single_ERR.log
        └── flight_request_status.json
```

### Files

#### `SignoffFlightsDataset_Single_<request_id>.parquet`

Single-flight dataset generated from the API call.

This file represents the raw single-line output before the final inference preprocessing step.

#### `API_Single_ERR.log`

Technical log file generated during the single-flight API pipeline.

It is used by the FlyOnTime team to investigate API errors, unavailable services, invalid flight requests, or unexpected pipeline failures.

#### `flight_request_status.json`

Structured status file generated for each request.

It is used by the FastAPI service to return clear user-facing messages to the Streamlit interface.

It can contain information such as:

- request status;
- error code;
- user-friendly error message;
- warning message;
- requested flight number;
- matched flight number;
- departure and arrival airports;
- generation timestamp.

## `processed/`

The `processed/` folder stores the cleaned and transformed files used by the inference service.

It corresponds to the **Silver layer** of the data lake.

Its structure mirrors the `raw/` folder.

### Structure

```text
processed/
└── YYYY-MM-DD/
    └── request_id/
        └── single_flight_model_input_<request_id>.parquet
```

Example:

```text
processed/
└── 2026-04-22/
    └── requete_AF7421_20260422_093538/
        └── single_flight_model_input_requete_AF7421_20260422_093538.parquet
```

### File

#### `single_flight_model_input_<request_id>.parquet`

Feature-engineered single-flight input generated after transformation.

This is the final processed file used by the inference service to align the live API data with the model input schema.

## Data lake logic

The FlyOnTime S3 storage follows a simple data lake organization:

```text
datasets/      → historical and training datasets
raw/           → raw single-flight outputs from API calls
processed/     → transformed inference-ready datasets
ppml-mlflow    → MLflow model artifacts
```

This structure makes the project easier to debug, reproduce, and explain.

It also allows each user request to be tracked independently through a unique `request_id`, which makes the architecture compatible with multiple simultaneous users.
